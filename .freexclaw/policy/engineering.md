# Engineering Policy

## 目标

默认将代码任务交付为“可运行、可维护、可验证”的工程化结果，而不是只生成单个演示文件。

## 默认原则

1. 优先遵循当前仓库已有结构、命名、目录层级和依赖管理方式。
2. 若当前目录为空或结构不足，则按目标语言/框架建立最小合理工程骨架。
3. 不把全部逻辑塞进单文件，除非用户明确要求。
4. 写完代码后，默认继续完成初始化、依赖安装、基础校验和构建验证。
5. 最终结果必须明确说明修改了什么、执行了什么、通过了什么、还缺什么。

## 文件层面

- 优先补齐入口文件、依赖清单、配置文件和必要目录
- 涉及服务或应用时，优先考虑分层结构而不是平铺实现
- 与现有项目集成时，尽量复用已有模块、配置和脚本

## 命令层面

- 优先执行可结束的安全命令，如安装、测试、lint、vet、build
- 避免默认启动长时间不退出的前台进程
- 若命令失败，应优先根据错误继续修复并重试
- 命令应限制在当前工作区内执行

## 语言最低交付标准

### Go

- 确保存在 `go.mod`
- 补齐依赖并执行：
  - `go mod tidy`
  - `go test ./...`
  - `go vet ./...`
  - `go build ./...`

### Node.js / TypeScript

- 确保存在 `package.json`
- 补齐依赖并尽量执行：
  - `npm install`
  - `npm test`
  - `npm run build`

### Python

- 确保存在 `requirements.txt` 或 `pyproject.toml`
- 补齐依赖并尽量执行：
  - `python -m pip install -r requirements.txt`
  - `pytest`
  - `python -m compileall .`

## 最终回复要求

最终结果默认应包含：

- 修改/新增文件
- 实际执行的命令
- 通过的校验
- 未通过或未验证的部分
- 用户下一步如何运行或继续
