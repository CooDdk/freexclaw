# 实时查询路由层设计

日期：2026-07-03
项目：FreeX Claw
范围：第一阶段实时信息能力增强，覆盖天气、新闻、金融

## 目标

增强 `web_search`，让 FreeX Claw 在回答实时问题时更像一个具备专用数据通道的助手，而不是单纯依赖普通搜索结果页。

第一阶段覆盖：

- 天气
- 新闻
- 金融
- 汇率
- 股价

本设计还需要为后续扩展航班、火车、地铁、公交等出行类能力预留清晰的扩展路径。

## 问题背景

当前实现会把所有实时查询都送进同一条通用网页搜索路径。对于天气、新闻、汇率、股价这类用户期望得到“结构化、当前、可直接回答”的问题，这条路径效果较差。

已观察到的失败模式：

- 通用搜索对天气和天气预报类问题容易返回低相关网页，比如百科、旅游页、门户首页。
- 像“未来7天的天气呢”这样的追问会丢失上一轮里的主题和地点信息。
- 系统无法区分“高可信的结构化实时数据”和“从搜索结果拼出来的低可信答案”。
- 继续在 `web_search` 里追加零散判断逻辑，扩展到更多实时领域后会越来越难维护。

## 推荐方案

在 `web_search` 和现有 Bing 回退逻辑之间增加一层“实时查询路由器”。

推荐流程：

1. 用户发送原始问题。
2. 程序在发送前先判断该问题是否属于已支持的实时领域。
3. 如果命中实时领域，程序结合当前会话上下文补全缺失信息，并生成规范化查询。
4. `web_search` 接收规范化后的查询。
5. 路由器将查询分发给对应领域的 provider。
6. provider 返回统一结构的实时结果。
7. 上层将该结果格式化成现有工具输出风格。
8. 如果路由失败或 provider 取数失败，则回退到通用网页搜索。

这样既能保留当前工具接口，又能减少模型在实时问题上的自由改写。

## 非目标

第一阶段不做以下内容：

- 新增 `<weather>`、`<live_query>` 之类的新工具标签
- 一次性覆盖航班、火车、地铁、公交
- 长周期、复杂对话记忆
- 每个领域接入多个数据源并做复杂排序
- 大规模终端 UI 改版

## 总体架构

### 1. 实时查询路由器

在 `internal/tools` 下增加一层路由逻辑，由 `WebSearch` 调用。

职责：

- 将查询分类为以下之一：
  - `weather`
  - `news`
  - `finance`
  - `generic_search`
- 从用户查询中提取结构化槽位
- 从最近一次实时查询上下文中补齐缺失槽位
- 将查询分发给对应 provider
- 决定返回结构化结果还是回退搜索

第一阶段的路由逻辑应当保持“确定性、轻量、可测”，优先使用规则和模式匹配，而不是再引入模型判断。

### 1.1 发送前预判

对于已支持的实时查询，程序应在消息发送后、模型生成前优先做一次预判。

目的：

- 避免模型把 `未来7天的天气呢` 改写成不稳定的站点搜索词
- 对天气追问直接生成稳定查询，例如 `武汉 7天 天气预报`
- 将“是否调用工具”和“工具使用什么查询词”尽量收回到程序侧控制

这一步属于第一阶段实现中的关键收口策略。

### 2. 实时查询上下文

增加一个会话级的小型上下文对象，只保存“最近一次成功实时查询”的结构化信息。

建议字段：

- `Domain`
- `Location`
- `TimeRange`
- `Topic`
- `Symbol`
- `MarketType`
- `ResolvedAt`

用途：

- 支持短追问，例如：
  - `未来7天的天气呢`
  - `那北京呢`
  - `今天的呢`
- 避免重新扫描整段历史消息或解析旧回复正文

规则：

- 只有成功完成的实时查询才能更新上下文。
- 上下文必须带领域信息，天气上下文不能污染金融查询。
- 当领域变化时，默认不跨领域继承槽位。第一阶段保持保守。

### 3. 领域 Provider

每个已支持领域都实现独立 provider，并保持稳定边界。

建议接口：

```go
type LiveProvider interface {
    Domain() string
    Match(query string, ctx LiveQueryContext) MatchResult
    Resolve(match MatchResult, ctx LiveQueryContext) (LiveResult, error)
}
```

如果实现时不完全照这个接口落地也可以，但必须保留以下职责分离：

- 路由判断
- 上下文补全
- 数据获取
- 结果格式化

### 4. 统一结果结构

所有 provider 都返回统一的 `LiveResult`。

建议字段：

```go
type LiveResult struct {
    Domain        string
    Title         string
    Summary       string
    Items         []LiveItem
    ResolvedQuery string
    SourceName    string
    SourceURL     string
    ResolvedAt    string
    Confidence    string
    FallbackHint  string
}
```

建议条目结构：

```go
type LiveItem struct {
    Label  string
    Value  string
    Detail string
}
```

可信度取值：

- `high`：直接来自结构化实时数据源
- `medium`：来自半结构化结果或 provider 整理结果
- `low`：来自搜索结果整理或信息不完整的回退路径

第一阶段里，`LiveResult` 再被转换成现有 `SearchResult` 风格的文本输出，这样工具接口和界面都不需要先改。

## 各领域设计

### 天气 Provider

第一阶段能力：

- 当前天气
- 今日概况
- 未来 `N` 天天气预报，至少支持 7 天请求

需要处理的槽位：

- `Location`
- `TimeRange`
  - `today`
  - `current`
  - `N-day forecast`

示例：

- `看一下今天武汉的天气情况`
- `未来7天的天气呢`
- `那北京呢`

期望输出字段：

- 天气状况
- 当前温度
- 体感温度
- 湿度
- 风速
- 每日最低和最高温
- 多日预报列表
- 数据更新时间

实现说明：

- 复用现有 Open-Meteo 接入。
- 将当前天气逻辑扩展到支持未来多日预报。
- 对天气类短追问，优先从上下文继承地点和领域。

### 新闻 Provider

第一阶段能力：

- 最近新闻摘要
- 指定主题新闻摘要
- 宽泛的“今天有什么新闻”类汇总

需要处理的槽位：

- `Topic`
- `Region`
- `TimeRange`

期望输出字段：

- 标题
- 来源
- 发布时间
- 链接
- 简短摘要

行为要求：

- 优先使用结构化或半结构化新闻来源。
- 如果直接结构化取数不稳定，可以退回到“精选搜索结果整理”，但必须在结果里明确标识这是回退路径。
- 不能把搜索拼接结果伪装成高可信实时新闻源。

### 金融 Provider

第一阶段先支持两个子类：

- 汇率
- 股票行情

需要处理的槽位：

- `Symbol` 或资产名称
- `MarketType`
  - `fx`
  - `equity`
- 汇率场景下的目标币种

示例：

- `美元兑人民币`
- `英伟达股价`

期望输出字段：

- 当前价格或汇率
- 涨跌额
- 涨跌幅
- 币种
- 市场或交易所
- 行情时间

行为要求：

- 通过清晰规则区分汇率和股票查询。
- 在进入取数前先完成子领域判断，不要把二者混在一条逻辑里。

## 查询分类

第一阶段采用规则式分类。

天气信号词：

- `天气`
- `气温`
- `温度`
- `降雨`
- `降水`
- `湿度`
- `风力`
- `预报`
- `未来7天`

新闻信号词：

- `新闻`
- `头条`
- `热点`
- `最新消息`
- `今天有什么`

金融信号词：

- `汇率`
- `兑`
- `股价`
- `股票`
- `价格`
- 已知 ticker 或常见资产名称

分类时需要同时考虑：

- 显式关键词
- 依赖上下文的短追问
- 查询是否不完整但可以从上下文安全补全

## 上下文补全规则

第一阶段应保持上下文继承逻辑克制、可预测。

规则：

1. 当前句子如果能明确命中某个实时领域，优先采用当前句子的领域判断。
2. 当前句子如果比较短、像追问、且可与上一条成功实时查询在同领域拼接，就进行补全。
3. 如果补全后仍缺少必要槽位，不要强行猜测，应进入回退路径。

示例：

- 上一轮：`今天武汉天气`
  当前轮：`未来7天的天气呢`
  解析：继承 `武汉`，并把时间范围设为 `7-day forecast`

- 上一轮：`今天武汉天气`
  当前轮：`那北京呢`
  解析：仍然属于 `weather`，地点替换为 `北京`

- 上一轮：`英伟达股价`
  当前轮：`今天的呢`
  解析：只有在规则仍能判断这是金融追问时，才继续沿用 `finance`

## 回退策略

建议使用三层回退机制。

### 第一层：领域内回退

如果 provider 已识别领域，但主数据源失败，则允许在该领域内部走安全的备用路径。

示例：

- 天气 provider 切换到备用天气获取方式
- 新闻 provider 回退到精选搜索结果整理

### 第二层：部分结果返回

如果只拿到部分数据，也应当返回，但必须明确告诉上层“数据不完整”。

示例：

- 用户请求 7 天天气，只拿到 3 天
- 股票行情拿到了当前价，但没拿到涨跌幅

系统不能把不完整数据伪装成完整答案。

### 第三层：通用搜索回退

如果路由器无法分类，或者所有 provider 都失败，则回到原有 Bing 搜索路径。

这样可以保持当前系统的最后兜底能力。

## 输出格式

第一阶段保持外部工具协议不变。

也就是说：

- `web_search` 仍然输出文本
- 结构化实时结果要被渲染成高信号的搜索式结果
- 格式中要带上时间、来源、可信度或回退提示

格式化优先级：

- 先给主答案
- 明确更新时间
- 标明来源
- 各领域字段尽量稳定一致
- 不完整或回退结果要显式提示

界面展示约束：

- 成功的 `web_search` 结果明细默认可以隐藏
- 最终正文末尾保留简短数据来源即可
- 工具失败时必须保留错误明细，便于定位问题

## 测试策略

### 路由测试

至少覆盖：

- `今天武汉天气`
- `未来7天的天气呢`
- `那北京呢`
- `今天有什么大新闻`
- `美元兑人民币`
- `英伟达股价`

### 上下文测试

至少验证：

- 天气追问能继承地点
- 天气追问能替换地点
- 金融追问不会误继承天气槽位
- 默认不允许跨领域继承

### Provider 格式化测试

至少覆盖：

- 当前天气结果
- 7 天天气结果
- 新闻摘要结果
- 汇率结果
- 股票结果

### 回退测试

至少验证：

- provider 失败时能正确回退
- 数据不完整时会明确标识
- 未知实时查询仍然可以进入通用搜索

## 实现落点建议

建议文件划分：

- `internal/tools/live_router.go`
- `internal/tools/live_context.go`
- `internal/tools/live_types.go`
- `internal/tools/weather.go`
- `internal/tools/news.go`
- `internal/tools/finance.go`

现有天气代码建议保留并吸收进 provider 结构，而不是整块推倒重写。

## 风险

- 新闻和金融数据源的一致性通常不如天气。
- 如果上下文继承过于激进，追问识别容易误判。
- 输出格式如果没有显式暴露不确定性，模型就容易“装确定”。
- 当前仓库的 `.gitignore` 会忽略源码目录，后续实现改动在 `git status` 里可能不可见，除非单独调整。

## 结论

第一阶段采用“路由器 + provider”的方式，挂在现有 `web_search` 背后，是当前仓库最稳妥的演进路径。

这样做能带来：

- 更好的实时答案质量
- 更好的追问承接能力
- 更清楚的数据来源透明度
- 更自然的后续扩展路径
